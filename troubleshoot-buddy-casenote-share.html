01 · WHY + THINKING

为什么要做这两个东西，以及背后的思考逻辑

不是为了"用 AI 而用 AI"，是把两个真实痛点解决掉 —— 一个是"模仿专家的脑回路"，一个是"把人工整理流水线化"。

A

Troubleshoot Buddy — 解决"思路卡壳"的痛

SCENARIO · 使用场景
当你做 Critical Sit、面对一个自己不熟的方向，
一时半会儿不知道该问客户什么、该看什么的时候 ——
让 troubleshootBuddy 帮你 30 秒内理清排查思路。
🧊

思路卡壳

复杂 case 涉及多个产品线，遇到经验盲区时问题问不到点上，时间一分一秒过去客户在催。

⏱️

Critical Sit 压力大

客户在等，没时间慢慢翻 KB、问同事、做实验，必须立刻 narrow down 出最可能的 1–2 个方向。

🧩

零散问几句不成体系

想到哪问哪，容易漏掉服务端、漏掉网络层，事后才发现一开始就问错方向，全部重来。

🧠

Troubleshoot Buddy 的思考逻辑

核心一句话：不是让 AI 更聪明，而是把"专家怎么想"显式地写成规则。

01

角色 = 思考边界

给 agent 钉死角色：Senior MS TSE，专精 Windows / Azure / M365 / Networking / Security。角色一旦明确，它就不会跨界乱答、不会用消费级口吻，回答风格自然收敛。

Role-First专业一致
02

分层 = 流程铁律

所有 C/S 问题都强制 服务端 → 网络 → 客户端 三层顺序。先确认"服务还活着"，再查其他一切。这是 TSE 多年沉淀下来的铁律，AI 不能跳层。

Server FirstNo Skip
03

概率 = 透明思考

每个可能原因都标概率，新证据进来就重新打分，错了立刻 pivot。思考过程可追溯、可挑战、可纠错，不是黑盒。

ProbabilityPivot Fast
🤔 那直接问 ChatGPT / Copilot 不行吗？

❌ 直接问通用 AI

回答风格随机，有时长篇大论，有时点到为止
想到哪问哪，可能上来就让你查客户端
答案没有固定结构，要自己消化二次组织
命令不一定能直接跑，没考虑收集输出
你给反馈后它不一定"更新结论"
没有 Microsoft 内部 KB / Wiki 检索能力

✅ Troubleshoot Buddy

固定 Senior MS TSE 角色，专业一致的口吻
强制 服务端 → 网络 → 客户端 三层提问
固定输出：澄清问题 + 诊断命令 + 概率排序
每条命令 copy-paste 即可执行，自带 Out-File 收集
你喂证据 → 它自动更新概率、收窄方向
挂载 CSS Wiki / Internal KB / MS Learn 等知识源
💡 我不是在造一个"更聪明的 AI"，我是把一个 Senior TSE 的排查脑回路写成了 prompt，让它每次都按这个套路工作。
B

CaseNote — 解决"邮件一长就抓瞎"的痛

SCENARIO · 使用场景
当你和客户来回邮件 20+ 封、
自己都记不清做过什么、要写 case note 交接或汇报的时候 ——
一行命令、30 秒生成结构化笔记。
📬

邮件太长翻不动

几十封 thread 来回找证据，半小时过去笔记还没动笔，关键时间点容易漏掉。

📂

本地证据散乱

TSS 分析、Procmon 笔记、ETL 摘要分散在 case 文件夹里，要手动拼起来很费时间。

📐

格式不统一

每次写的结构都不一样，自己回头看吃力，交接给同事或者发给 manager 更费劲。

💡

CaseNote 的思考逻辑

核心一句话：不要 AI 自由发挥，要把人工整理动作拆成可重复的流水线。

01

触发 = 确定性

只在 /casenote <id> 显式调用时跑。不"猜"用户意图、不"顺手"生成。一个命令对应一种产出，边界清晰。

ExplicitDeterministic
02

多源融合 = 单一来源不够

邮件只有客户视角，本地日志只有技术视角。邮件 + 本地分析文件同时喂进来，AI 才能给出既有上下文又有技术深度的笔记。

Mailbox + FilesDual Source
03

模板 = 可交接的资产

固定三段：Issue / Troubleshooting / Status。今天给自己看，明天交接给同事，下周回头复盘 —— 格式一致，不用每次重新理解。

3-SectionHandover-Ready
02 · HOW IT WORKS

实现逻辑

露一点核心内容，方便理解 —— 但不贴代码，全是"东西怎么串起来的"。

🧠

Troubleshoot Buddy Agent

一个 .agent.md 文件 = 角色 prompt + 工作流 prompt + 工具挂载
⓵ 角色定义
Senior Microsoft TSE

专精领域显式列出来：

  • Windows：DS / UEX / SHA / Net / Perf / Sec / Storage / Deploy
  • Azure：VM / Network / Identity / AKS
  • M365：Exchange / Teams / Intune / Entra
  • 跨产品：Wireshark / TLS / PKI
⓶ 三阶段工作流
Phase 1 / 2 / 3
  • Phase 1：初始评估 — 出澄清问题 + 诊断命令 + 初步概率
  • Phase 2：迭代诊断 — 收到反馈就更新概率、标 ❌ 或 ✅
  • Phase 3：根因 + 修复 — 给出 fix / workaround / rollback
⓷ 分层提问铁律
Server → Network → Client

所有 C/S 问题强制按这个顺序问。内置一张「第一个该问的服务」速查表，比如：

  • DFS 故障 → 先查 Dfs
  • RDP 故障 → 先查 TermService
  • 打印故障 → 先查 Spooler
⓸ 命令质量规则
直接可执行 + 自带收集
  • 没有占位符，copy-paste 即可跑
  • 每条命令都加 | Out-File 落盘
  • 统一输出目录 $env:USERPROFILE\Desktop\Diag_*
  • 方便客户一键打包回传
⓹ 视觉标记
概率一目了然

固定一套 emoji 语义：

  • 🔴 高概率 / 🟠 中 / 🟡 低 / ⚪ 可能
  • ❌ 已排除 / ✅ 已确认
  • 表格 + 简易 ASCII 图，避免长篇大论
⓺ 挂载知识源
不是孤岛 AI
  • CSS Wiki — 内部 TSG / SOP
  • Internal KB — 已发布 KB
  • OSBugs — 已知 bug 库
  • MS Learn — 公开文档
你输入症状 → 分层提问 + 命令 + 概率 → 你喂反馈 → 更新概率 / pivot → 根因 + 修复
📝

CaseNote Skill

一个 skill = 触发条件 + 工作流步骤 + 输出模板
⓵ 严格触发
只认 /casenote <id>

用户必须显式带斜杠命令才触发。聊 case 进展、问"总结一下"都不会被误触发。

  • 必填：case id（默认 16 位）
  • 可选：本地日志路径
  • 可选：输出根目录
⓶ 工作目录策略
每个 case 一个文件夹

在输出根目录下查找以 case id 开头的子文件夹，找到就复用，没有就新建：

  • 默认输出根目录：C:\temp\Cli_files
  • 查找：根目录下名字以 <case id> 开头的子文件夹
  • 新建：C:\temp\Cli_files\<case id>\
  • 自定义：调用时加 output: D:\xxx 改根目录
  • 根目录不存在时自动创建（不报错、不要求手动准备）
⓷ 邮件抓取顺序
Sent Items 优先
  • 先搜 Sent Items（你的视角最全）
  • 用 DASL filter 按 subject 包含 case id
  • 按时间正序，单条 body 截断到 5000 字
  • Sent Items 为空才回退 Inbox
⓸ 本地证据合并
用户指定路径才读
  • 只读用户传入的文件 / 文件夹
  • 支持 .txt / .md / .log / .csv / .json / .xml / .html
  • 大文件按关键字段优先读
  • 不动用户源文件
$ /casenote 2606010040001261
只抓邮件 · 不读任何本地文件
$ /casenote 2606010040001261 log: C:\temp\Cli_files\xxx\analysis.md
抓邮件 + 读你指定的 analysis.md
⓹ 证据融合规则
新证据优先 + 冲突标注
  • 按时间排序，新 > 旧
  • 邮件 vs 本地分析冲突 → 在 Status 段标注
  • 缺失字段写 Not identified from available data
  • 不编造、不脑补
⓺ 固定输出模板
三段纯文本
  • Issue Definition — 症状 + 环境 + 影响
  • Troubleshooting Done — 时间线 + 动作 + 证据
  • Current Status — 根因 + 下一步 + 负责人
  • 文件名：case_note_summary_<时间戳>.txt
/casenote <id> → 找/建工作目录 → 抓邮件 + 读本地 → 融合 + 去重 → 三段模板 → .txt 落盘

04 · HOW TO USE

怎么用

从拿到 case 到归档，三步走，每步对应一个能力。

STEP 01 · 拿到新 case

🚀 启动 Troubleshoot Buddy

在 Clawpilot 里 @TroubleshootBuddy，把客户报的现象一句话扔给它，立刻拿到 分层澄清问题 + 可执行诊断命令 + 初步概率排序。

@TroubleshootBuddy 客户 DFS 0x80070035

💻 如果用的是 CLI：先跑 /agents，然后在下拉列表里选 TroubleshootBuddy 即可。

/agents → ↓ 选择 TroubleshootBuddy
STEP 02 · 排查中

🔁 边查边喂证据

客户回了日志、命令有了输出 —— 直接贴回去。Buddy 会 更新每个可能原因的概率，标记已排除的方向，给出下一步收窄的命令。

"这是客户的事件日志，看看怎么解读"
STEP 03 · 收尾归档

📝 一键生成 Case Note

case 处理告一段落，调用 CaseNote 自动抓邮件 + 合并本地日志，输出 结构化 .txt，可直接粘到 case 系统、邮件、交接文档。

/casenote 2606010040001261

05 · HOW TO GET

怎么装到你自己的环境

两个东西都已经发布到内部 cnwin plugins 仓库，一行命令就能装。

github.com/gim-home/cnwin/tree/main/plugins

这个仓库里收录了 cnwin 团队的所有 Copilot plugins（包括 troubleshoot-buddy、pcap-analysis、casenote 等）。 你只需要一行 copilot plugin install 就能把对应 plugin 拉到本地。

▶ 安装 TROUBLESHOOT BUDDY
$ copilot plugin install gim-home/cnwin:plugins/troubleshoot-buddy
▶ 安装 CASENOTE
$ copilot plugin install gim-home/cnwin:plugins/casenote
1

2

📦 Plugin 命名规则

所有 plugin 都遵循 copilot plugin install gim-home/cnwin:plugins/<plugin-name> 格式。
比如装 pcap 分析工具就是 copilot plugin install gim-home/cnwin:plugins/pcap-analysis。
去 仓库 里看到喜欢的，把 plugin 文件夹名替换上去就行。
