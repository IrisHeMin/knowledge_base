1️⃣ 起源 —— 你最初的想法

你最初提出的需求很简洁，但信息量非常密集。我把它拆解为五个"需求维度"：

维度你的原话 / 隐含意图设计含义
主题"给大家普及一些有用的方法论"，举例 STAR 写年终总结选材要覆盖思考/表达/执行/复盘四大类，且要真正能落地
场景"很多时候不会处理不是不会，而是没有系统方法"每个方法不能只给名字，必须给"什么时候用"+"怎么用"
调性"欢乐地参与的同时，能有所收获"不能像培训，要像游戏；不能纯娱乐，要有干货沉淀
形式"做一个小型的 app 或者类似的方式"需要互动载体，不能 PPT；倾向 Web 而非原生
价值主张"体现我们与 AI 与时俱进"AI 必须深度融合到玩法里，不是噱头

后续在第二轮对话，你又补了一条关键约束：

⚠️ 关键约束：I 人友好
"我希望可以尽量让大家有参与感，不至于冷场，毕竟团队里面是有很多的 I 人。"
这一句直接锁死了后面所有交互设计的方向 —— 不能是"抽到谁谁必须现场即兴"的硬模式。

2️⃣ 构思阶段 —— 我是怎么想的

2.1 先看历史：你过去办的活动什么风格

我做的第一件事不是动手写代码，而是先调用 Outlook 工具，从你的 Sent Items 里搜 "idea roaming"，看历史几期是怎么办的：

  • 第 2-4 期：午餐 + 奶茶 · 沪/锡双地连线 · 11:30-13:00
  • 第 4 期：已经用 AI 做过 HTML 网页 → 说明你团队对 web 形态接受度高
  • 整体调性：轻松、午休时段、跨地连线

这给了我两个锚点：① 时间窗只有 ~1.5 小时，环节不能太多；② 用 HTML 网页有先例，不用解释技术形态。

2.2 提炼核心隐喻："方法论盲盒"

三个名字候选过：

  • ❌ "方法论小课堂" —— 太像培训，违反"游戏化"
  • ❌ "AI 共创工作坊" —— 太务虚，焦点跑到 AI 上
  • ✅ "方法论盲盒" —— 既有"惊喜感"（盲盒），又有"实物感"（每个方法像可收集的小卡片）

2.3 抛出初版玩法 + 用 ask_user 锁定方向

构思了"4 大类 × 扭蛋 × 场景 PK × AI 同台"组合后，没有直接动手写，而是用 ask_user 工具问你 4 个关键问题：

  1. 主题方向：要 4 大类混合 还是 单一类深挖？
  2. 玩法：扭蛋 / 场景 PK / AI 同台 三选哪些？
  3. AI 怎么实现：实时调 API 还是预置答案？
  4. 需不需要附带邀请邮件？
💡 设计原则
在做大决策前用 ask_user 一次问清楚，比写完了改三遍快。后来这个习惯保留了 —— 双语化范围、PWA 分发方式都用了 ask_user。

3️⃣ 技术选型 —— 为什么是单文件 HTML + PWA

3.1 形态选择：为什么不用 React / Vue / 原生 App？

方案优点致命缺点判断
原生 App (iOS/Android)体验最好需要双平台开发 + 应用商店审核 + 公司证书❌ 完全不可行
Electron 桌面 App独立窗口包大小 100MB+、跨端要打三个版本❌ 用户也是这意见
React/Vue + 后端工程化好需要服务器、npm 依赖、构建工具链❌ 杀鸡用牛刀
单文件 HTML (vanilla JS)零依赖、零构建、邮件能附件发代码可读性差✅ 初版选这个
+ PWA 升级可"装到桌面"、能离线需要 HTTPS 托管✅ GitHub Pages 解决

3.2 部署选择：为什么是 GitHub Pages

  • 你已经有 IrisHeMin/knowledge_base 这个 Jekyll 站点，账号、Pages 配置都现成的
  • HTTPS 自动配置（PWA 必须 HTTPS）
  • 免费 · 永久托管 · 团队成员一个 URL 直接访问
  • 但你后来说"不想暴露 knowledge_base" → 新建了独立 repo idea-roaming-5

3.3 渲染策略：服务端 vs 客户端

全部数据（24 个方法、18 个辩题、36 组 AI 救场论据）直接以 JavaScript 字面量写在 HTML 里，没有任何 fetch / API 调用。原因：

  • 数据量小（~130KB），一次性下载完毕
  • 离线可用（Service Worker 缓存）
  • 无 API key 泄露风险（不调外部 AI）
  • "AI 救场"是预生成的论据库，不是实时调用 —— 你最初就选了"预置 GPT 答案"

5️⃣ 关键技术点深挖

5.1 AI 救场的三次重构 —— 最值得拿出来讲的一段

这是整个项目里花时间最多、推翻最多次的部分。三版的核心差异：

版本实现方式问题
v1每个方法硬编码一个 sample 答案跟当前辩题完全无关 —— 你截图"用 SCQA 辩 AA 制，给出的却是周报必须写的内容"

✅ 最终方案：DEBATE_ARGS 数据结构

5.2 辩论方法收窄：从 12 → 5

原本 12 个方法都可以抽来辩论，但你试用后发现"奥卡姆剃刀 / 5 Why / 第一性原理"在辩论场景毫无用武之地。我做了筛选：

  • ✅ 保留 5 个表达类方法：金字塔 / SCQA / PREP / STAR / 黄金圈 —— 都是"先抛结论再展开"的论证型结构
  • ❌ 剔掉 7 个硬思考方法：5 Why / MECE / 奥卡姆 / 第一性原理 / SWOT / 5W1H / FFC —— 这些要现场列因果链，模板套不出来

但代码里 12 个的生成器都保留着（dead code），以备未来扩回。

5.3 Service Worker 策略调整

SW 缓存策略经过两次迭代：

  • v1（cache-first 全包）：所有请求先查缓存 → 用户永远看到旧版，必须手动清缓存才能更新 ❌
  • v2（混合策略）：HTML / 导航请求走 network-first（先拿新的，离线才退回缓存）；静态资源 (PNG/SVG/CSS) 走 cache-first（极少变） ✅
self.addEventListener('fetch', e => {
  const isHTML = e.request.mode === 'navigate' ||
                 (e.request.headers.get('accept') || '').includes('text/html');
  if (isHTML) {
    // network-first: updates show up immediately on refresh
    e.respondWith(fetch(e.request).catch(() => caches.match('./index.html')));
  } else {
    // cache-first for static assets
    e.respondWith(caches.match(e.request).then(hit => hit || fetch(e.request)));
  }
});

5.4 Outlook 暗色模式踩坑

Newsletter 第一版用深紫色 Hero + 黑色 App 召唤区。在 Outlook 暗色模式下 Outlook 会"自动反色"试图救场，结果：

  • 白字被反色成深色 → 跟原本就深色的背景叠加 → 完全看不到

解决方案：整封邮件全部用浅色背景 + 深色文字，accent 只用饱和度高的橙黄（暗模式反色后仍有对比）。

6.3 决策原则：什么时候用什么

📋 我的工具选择逻辑
  • 能 view/edit 解决就不写 powershell：单点的字符串改动 → edit；批量重命名或要执行逻辑 → powershell
  • 能 grep 就不 Get-Content + Select-String：grep 跨平台、对大文件更快
  • 大决策一定 ask_user：减少返工。这个项目我估计 ask_user 至少救了 2-3 次大改
  • 能在本地解决就不调外部 API：图标用 System.Drawing 生成 vs 调 OpenAI DALL-E；前者零成本零延迟
  • 能用现成账号就不要新建：先想到用你的 IrisHeMin GitHub 账号 + 现成的 Pages 配置

9️⃣ 复盘 —— 哪里走过弯路

9.1 弯路清单

  1. JS 字符串转义错误炸全场：单引号嵌套写错一处导致整个 script 块不执行，5 个 tab 全废。
    → 教训：每次大改后必须跑 node --check
  2. AI 救场绕弯子绕了三圈：以为方法越多越好、模板套得越严越好。实际用户要的就是"给我一段能念的话"。
    → 教训：先听用户原始诉求，别自作主张抽象
  3. PWA 上来就用 cache-first：导致用户看不到任何更新。
    → 教训：HTML 永远应该 network-first
  4. manifest 写 portrait orientation：桌面也按手机尺寸开窗 → 用户截图反馈窗口奇怪。
    → 教训：orientation 对桌面端是干扰项
  5. Newsletter 用深色背景：Outlook 暗色模式破坏可读性。
    → 教训：邮件 HTML 永远首选浅色高对比度
  6. Jekyll 重复部署污染 knowledge_base：把 PWA 放在私人 knowledge_base 子目录 → 你不想暴露。
    → 教训：项目应该有独立的部署目录

9.2 做对的几件事

  • ✅ 开始之前 ask_user 锁方向 —— 省了至少两次大返工
  • ✅ 用 Outlook 看历史活动风格 —— 调性一开始就对了
  • ✅ 渐进式增强：先单文件 HTML 跑通 → 再 PWA 化 → 再 Newsletter，每步可单独发布
  • ✅ 数据和渲染分离：24 方法 / 18 辩题 / 36 论据都是数据对象，渲染靠模板字符串，改文案不动 UI
  • ✅ 每个决策都说明 why：你能看懂"为什么这么做"，下次想自己改也容易

9.3 如果重来一次，会怎么改进

  • 第一版就上 PWA，省得后面再补 manifest/sw
  • AI 救场直接做 v3 版本（论据库），跳过 v1/v2
  • 独立 repo 一步到位，不要先放 knowledge_base
  • Newsletter 直接用浅色模板
🎯 总结一句话
